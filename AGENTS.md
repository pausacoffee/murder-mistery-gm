# 프로젝트 지침

## 목표
이 저장소는 100개 이상의 머더 미스터리류 게임을 지원하는 정적 GM 플랫폼을 목표로 합니다.

장기 목표는 다음과 같습니다.
- 서버 없이 클라이언트 측 게임 카탈로그 검색 제공
- 게임별 독립 패키지 구조 유지
- 재사용 가능한 슬라이드, 타이머, 규칙 렌더링 제공
- 스캔된 룰북으로부터 에이전트가 게임 패키지를 생성할 수 있는 구조 마련

## 데이터 원본 위치
- `data/catalog.json`: 검색 및 리스트 표시용 경량 카탈로그 데이터만 저장
- `games/<id>/game.json`: 게임 패키지의 진입점
- `games/<id>/slides.html` 또는 향후 `games/<id>/slides.json`: GM 진행용 슬라이드 데이터
- `games/<id>/rules.json`: 외부 분리된 규칙 모달 데이터
- `games/<id>/assets/`: 박스 이미지, 오디오 등 관련 에셋
- `styles/base.css`: 리셋, 공통 타이포그래피, 기본 유틸리티
- `styles/layout.css`: 뷰 단위 레이아웃, 프레임, 상단 바 같은 큰 구조
- `styles/components.css`: 카드, 타이머, 모달, 규칙 패널 등 재사용 컴포넌트
- `styles/responsive.css`: 반응형 대응만 담당하는 미디어 쿼리
- `styles/themes/default.css`: 모든 게임에 공통 적용되는 기본 테마
- `styles/themes/<theme>.css`: 선택적인 게임 전용 테마 오버라이드

전체 게임 본문 데이터를 다시 `data/catalog.json`에 넣지 마십시오.

## 패키지 규칙
- 모든 게임은 안정적인 `id`를 가져야 합니다.
- 모든 카탈로그 항목은 `gamePath`를 포함해야 합니다.
- 검색용 필드는 카탈로그에 둡니다: `name`, 인원 수, `synopsis`, `searchText`
- 실행용 필드는 게임 패키지에 둡니다: 슬라이드 경로, BGM 설정, 규칙 탭, 커스텀 동작
- `app.js`에 게임별 하드코딩 분기를 늘리기보다 선언형 데이터 구조를 우선합니다.
- 헤더의 추가 버튼은 고정 마크업으로 늘리지 말고, 게임 패키지의 `headerActions[]` 데이터로 선언합니다.
- 게임 패키지에 `theme`를 둘 수 있으나, 전용 테마 파일이 모든 게임에 반드시 필요한 것은 아닙니다.
- 전용 테마가 없을 때는 항상 `styles/themes/default.css`만으로 정상 표시되어야 합니다.

## 스타일 구조 규칙
- 새 스타일을 추가할 때는 먼저 `base / layout / components / responsive / themes` 중 어디에 속하는지 판단합니다.
- 공통 색상, 프레임 배경, 강조색처럼 테마에 따라 바뀔 값은 가능하면 CSS 변수로 두고 `default.css`에서 기본값을 정의합니다.
- 게임 전용 테마는 `styles/themes/<theme>.css`에만 작성하고, 공통 CSS 파일에 특정 게임명을 직접 넣지 마십시오.
- 게임별 전용 테마가 없더라도 UI가 온전히 동작해야 합니다. 즉, 기본 테마 fallback은 필수입니다.
- 단순한 레이아웃 수정 때문에 게임 전용 테마 파일을 만들지 마십시오. 공통 구조 수정은 `layout.css` 또는 `components.css`에서 처리합니다.

## 헤더 액션 규칙
- 게임별 헤더 추가 버튼은 `game.json`의 `headerActions[]`로 관리합니다.
- 각 항목은 최소한 `id`, `label`, `showOn`, `action` 구조를 가집니다.
- 버튼 배치는 항상 동적 렌더링을 우선합니다. 게임별 버튼 때문에 `index.html`에 고정 버튼을 추가하지 마십시오.
- 현재는 배치와 선언형 구조만 안정적으로 유지합니다.
- 액션 기능 타입이 늘어날 수 있으나, 새로운 기능이 실제로 필요해질 때 `action.type` 확장과 렌더러 리팩터링을 진행합니다.
- 즉, 미리 과도한 액션 타입 체계를 만들지 말고, 현재 필요한 범위까지만 구현합니다.

## 슬라이드 구조 방향
현재 전환 단계에서는:
- 하위 호환성을 위해 `slides.html` 사용을 허용합니다.

목표 구조는 다음과 같습니다.
- 신규 게임은 점진적으로 `slides.json` 기반으로 이동합니다.
- 슬라이드 정의는 `story`, `timer`, `rules`, `order`, `vote`, `selection`, `ending`, `custom` 같은 타입 기반 블록을 사용합니다.
- 타이머 정의는 `basic`, `per-player` 같은 모드를 갖는 데이터 기반 구조로 관리합니다.
- 페이즈 전환 확인 조건은 코드에 직접 박지 말고 데이터로 선언하는 방향을 우선합니다.

## 순서 다이어그램 규칙
- 순서 다이어그램이나 진행 흐름 표시에서 `→`, `=>`, `->` 같은 텍스트 화살표를 직접 표시하지 마십시오.
- 화살표는 반드시 CSS 또는 SVG 기반의 벡터 표현으로 구현합니다.
- 즉, 텍스트 노드에 화살표 문자를 넣는 대신 전용 요소와 스타일을 사용합니다.
- 기존 프로젝트의 `arrow` 스타일 패턴이 있으면 그 방식을 우선 재사용합니다.

## 룰북 변환 방향
스캔된 룰북을 게임 패키지로 변환할 때는 다음 순서를 따릅니다.
- 먼저 메타데이터를 추출합니다: 제목, 인원 수, 시놉시스
- 전체 규칙 탭 구성을 식별합니다.
- 파트 및 페이즈의 진행 순서를 추출합니다.
- 각 페이즈의 타이머 값과 공용 타이머인지 플레이어별 타이머인지 구분합니다.
- 파트 전환 조건과 확인 다이얼로그 필요 여부를 식별합니다.
- 마지막은 엔딩 슬라이드 또는 엔딩 안내 문구로 마무리합니다.
- 확실하지 않은 정보는 임의로 채우지 말고 placeholder로 남깁니다.

## 수정 원칙
- 이미 지원 중인 게임은 의도적으로 마이그레이션하는 경우가 아니면 기존 동작을 유지합니다.
- 게임별 예외 처리를 하나 더 추가하기보다, 재사용 가능한 스키마/로더 코드를 우선합니다.
- 특정 게임에만 필요한 예외 동작이 있다면 패키지 메타데이터나 전용 렌더러 경로로 격리합니다.
- 패키지 구조나 작성 흐름이 바뀌면 `README.md`도 함께 갱신합니다.

## 검증
리팩터링 후에는 다음을 확인합니다.
- `node --check app.js` 실행
- 카탈로그가 올바른 게임 패키지 경로를 가리키는지 확인
- 리스트 -> 상세 -> GM 진입 흐름이 정상 동작하는지 확인
